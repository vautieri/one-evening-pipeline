## 1. Purpose & Scope

### 1.1 Purpose

This Software Development Plan (SDP) defines the processes, methods, standards, and environment used to develop the SynthFlight UAV Mission Simulation and Verification Platform. It satisfies the DO-178C objective requiring a plan that describes the software development processes, including the software development environment, applied to produce the software product.

This plan is the governing document for **how** SynthFlight software is developed. All development personnel shall comply with the processes defined herein. Deviations require formal approval through the Problem Reporting process (Section 12) and concurrence from the Quality Assurance Lead.

### 1.2 Scope

This SDP applies to:

- All software developed for the SynthFlight product, classified at **Design Assurance Level A** (Catastrophic failure condition) per DO-178C.
- All software life cycle phases: requirements, design, implementation, integration, and verification.
- All COTS and open-source components integrated into SynthFlight, including JSBSim 6-DOF physics engine (v1.2.4).
- All platforms on which SynthFlight is built and executed: Windows 10+ (x86-64) and Ubuntu 22.04+ (x86-64).
- All CI/CD automation executing in the GitHub Actions environment.
- All artifacts generated by or with the assistance of LLM-based AI tools (Goose, Claude).

This SDP does not cover system-level (non-software) development activities, hardware development, or field-loadable software procedures.

### 1.3 DAL A Applicability

As a DAL A product, SynthFlight development is subject to the most stringent objectives defined in DO-178C Tables A-1 through A-10. Key DAL A implications on the development process include:

| DO-178C Aspect | DAL A Requirement |
|----------------|-------------------|
| Independence | Verification activities require independence from the developer |
| Structural Coverage | MC/DC (Modified Condition/Decision Coverage) required |
| Traceability | Bidirectional traceability: requirements <-> design <-> code <-> tests |
| Reviews | All review objectives apply, with independence |
| Derived Requirements | Must be identified, traced, and communicated to the safety process |
| Dead Code | Zero tolerance — all dead code shall be eliminated or justified |
| Deactivated Code | Requires full verification of all configurations |

## 3. Software Life Cycle Model

### 3.1 Model Selection

SynthFlight development follows an **iterative/incremental life cycle model** as recognized by DO-178C Section 4.0. Each iteration produces a verified increment of functionality that satisfies a defined subset of the SRS-SF-2026-001 requirements.

This model is selected because:

- The simulation domain benefits from progressive integration and early feedback.
- JSBSim integration and physics model validation require iterative tuning.
- Multi-domain requirement coverage (Defense, Commercial, GIS, Infrastructure, SAR) is best addressed in planned increments.
- The model supports CI/CD automation through GitHub Actions.

### 3.2 Life Cycle Phases

Each iteration traverses the following phases:

```
   Requirements ──> Design ──> Implementation ──> Integration ──> Verification
        │              │            │                  │               │
        │              │            │                  │               │
   [SRS Baseline]  [SDD/SDS]   [Source Code]    [Build/Link]    [Test Results]
        │              │            │                  │               │
        ▼              ▼            ▼                  ▼               ▼
    Review Gate    Review Gate  Review Gate       Review Gate     Review Gate
```

**Phase 1 — Software Requirements Process** (Section 6)
- Elicit, analyze, and document high-level and low-level requirements.
- Baseline requirements in SRS-SynthFlight.md.

**Phase 2 — Software Design Process** (Section 7)
- Produce high-level design (architecture, CSCI decomposition).
- Produce low-level design (detailed module/class design).
- Establish design-to-requirement traceability.

**Phase 3 — Software Coding Process** (Section 8)
- Implement source code in C++17 per SCS-SF-2026-001.
- Perform peer review with DAL A independence.
- Verify coding standard compliance.

**Phase 4 — Software Integration Process** (Section 9)
- Integrate modules bottom-up: math libraries, physics models, subsystems, application.
- Execute integration tests.
- Build on both target platforms (Windows 10+, Ubuntu 22.04+).

**Phase 5 — Software Verification Process**
- Execute verification activities per SVP-SF-2026-001.
- Achieve MC/DC structural coverage.
- Produce verification results and coverage reports.

### 3.3 Iteration Planning

| Increment | Scope | Target Requirements |
|-----------|-------|---------------------|
| I1 | Platform scaffolding, build system, test framework integration | REQ-SF-0100–0199 (Platform & Build) |
| I2 | Simulation engine core, JSBSim integration, atmosphere model | REQ-SF-0200–0299, REQ-SF-0300–0399 |
| I3 | Terrain engine, DTED/LiDAR/GeoTIFF support | REQ-SF-0400–0499 |
| I4 | Flight planning, autopilot, sensor modeling | REQ-SF-0500–0699, REQ-SF-0700–0799 |
| I5 | Airspace, threat modeling, scenario system | REQ-SF-0800–0999, REQ-SF-1000–1099 |
| I6 | Multi-vehicle, visualization, Vulkan rendering | REQ-SF-1100–1199, REQ-SF-1200–1299 |
| I7 | Verification harness, traceability, data recording | REQ-SF-1300–1499, REQ-SF-1500–1599 |
| I8 | Security, interoperability, performance, GUI | REQ-SF-1600–1999 |
| I9 | Final integration, system-level verification, certification evidence | All — regression and closure |

### 3.4 Iteration Gates

Each iteration concludes with a formal review gate. Progression to the next iteration requires:

1. All phase transition criteria satisfied (Section 10).
2. All open DRs classified Severity 1 or 2 resolved or dispositioned.
3. Traceability matrix updated and verified.
4. QA audit of iteration artifacts completed.
5. Gate review minutes signed by Program Manager and QA Lead.

## 5. Software Development Standards

| Attribute | Value |
|-----------|-------|
| Standard Document | SRS-STD-SF-2026-001 (Software Requirements Standard) |
| Applicability | All software requirements in SRS-SF-2026-001 |

The requirements standard defines:

- Requirement syntax rules (shall-statement format, unambiguous, verifiable, atomic).
- Requirement attributes: ID, title, description, origin, domains, priority, WBS, verification method.
- Numbering convention: `REQ-SF-NNNN` with category-based ranges (19 categories, see SRS Section 2).
- Criteria for well-formed requirements: complete, consistent, correct, verifiable, traceable.

### 5.2 Design Standard

| Attribute | Value |
|-----------|-------|
| Standard Document | SDS-SF-2026-001 (Software Design Standard) |
| Applicability | All design artifacts (HLD and LLD) |

The design standard defines:

- Architecture description conventions (component diagrams, interface specifications).
- CSCI decomposition rules and naming conventions.
- Data dictionary and data flow conventions.
- Class/module design templates.
- Design review checklists (completeness, correctness, consistency, traceability, verifiability).
- Constraints on use of C++17 features in design (permitted OOP constructs per DO-332 assessment).

### 5.3 Code Standard

| Attribute | Value |
|-----------|-------|
| Standard Document | SCS-SF-2026-001 (Software Code Standard) |
| Language | C++17 (ISO/IEC 14882:2017) |
| Compiler | GCC (version baselined per Section 4.2) |

The code standard defines:

- Naming conventions (files, classes, functions, variables, constants, macros).

- Exception handling policy.
- Header file structure and include guard conventions.
- Comment and documentation requirements (Doxygen-compatible).

- Rules for COTS/open-source code that may deviate from the project standard (JSBSim).

### 6.3 Derived Requirements Identification and Handling

A **derived requirement** is any requirement that is not directly traceable to a higher-level requirement (system requirement or domain source). Derived requirements arise from:

- Design decisions (e.g., internal data structures, buffer sizes).
- Implementation constraints (e.g., platform-specific behavior).
- Safety analysis (e.g., additional error handling identified during FHA/FMEA).

Derived requirements handling process:

1. Developer identifies a need for a requirement not traced to an existing source.
2. Developer creates the derived requirement in SRS-SynthFlight.md with `Derived: Yes`.
3. Developer notifies the safety assessment process (per ARP 4754A / ARP 4761).
4. Safety engineer assesses the derived requirement for safety impact.
5. QA reviews and approves the derived requirement.
6. Traceability matrix is updated to reflect the derived requirement origin.

### 6.4 Requirements Review and Approval Process

| Review Activity | Participants | Independence | Artifact |
|----------------|--------------|--------------|----------|
| Requirements peer review | Author + independent reviewer | Required (DAL A) | Review checklist, comments |
| Requirements inspection | Requirements team + QA + safety | Required (DAL A) | Inspection report |
| Requirements baseline approval | Program Manager, QA Lead, Safety Lead | N/A | Signed approval record |

Review criteria (per DO-178C 6.3.1):

- [ ] Requirements comply with SRS-STD-SF-2026-001.
- [ ] Requirements are accurate and consistent.
- [ ] Requirements are verifiable with defined acceptance criteria.
- [ ] Requirements are traceable to higher-level requirements or identified as derived.
- [ ] Requirements conform to the coding standard where implementation constraints are specified.
- [ ] Requirements are compatible with the target hardware/OS platforms.
- [ ] Algorithmic requirements are accurate (e.g., atmospheric models, physics equations).

### 6.5 Traceability from Requirements

Every requirement shall be forward-traced to:

- One or more design elements (SDS).
- One or more source code modules/functions.
- One or more verification procedures (test cases, analyses, or inspections).

This bidirectional traceability is detailed further in Section 11.

- CSCI boundaries and interfaces (API contracts).

### 7.2 Low-Level Design (Detailed Design)

The low-level design refines each CSCI into CSUs (Computer Software Units) — the smallest units of code that can be independently tested. For each CSU, the low-level design shall specify:

- Class or module definition (name, responsibility, collaborators).
- Public interface (method signatures with pre/post conditions).
- Algorithm descriptions for non-trivial logic.
- Data structures and their invariants.
- Error handling behavior.
- Resource usage (memory, file handles, GPU resources).
- Thread safety constraints.

| Review Activity | Participants | Independence | Artifact |
|----------------|--------------|--------------|----------|

- [ ] Design complies with SDS-SF-2026-001.
- [ ] Design is traceable to requirements (every requirement has a design element).
- [ ] Design is consistent (no contradictions between CSCIs).
- [ ] Design is verifiable (test strategy exists for every design element).
- [ ] Design correctly implements algorithms (atmospheric model, physics, navigation).
- [ ] Design addresses robustness (error handling, boundary conditions, resource limits).
- [ ] Design avoids unnecessary complexity.
- [ ] Design correctly uses C++17 features per DO-332 OOT assessment.

### 7.4 COTS Design Considerations

JSBSim v1.2.4 (open-source, C++, CMake) is the primary COTS component. Design activities for COTS integration shall include:

- Wrapper/adapter layer design isolating JSBSim from SynthFlight internals.
- Interface specification between CSCI-SIMENGINE and JSBSim.
- Identification of JSBSim functions used and their behavioral contracts.
- Error propagation strategy from JSBSim to SynthFlight error handling.
- Version pinning and change control of JSBSim source.

## 8. Software Coding Process

### 8.1 Coding Standards Compliance

All SynthFlight source code shall comply with SCS-SF-2026-001 (Software Code Standard), which mandates:

- **Language:** C++17 (ISO/IEC 14882:2017).
- **Compiler:** GCC (baselined version per Section 4.2).
- **Compliance verification:** Static analysis (clang-tidy, cppcheck) integrated into CI pipeline.
- **Deviations:** Any deviation from the code standard shall be documented with justification, reviewed by an independent reviewer, and approved by the QA Lead. Deviations are tracked as Problem Reports.

### 8.2 Code Review Requirements

For DAL A, code reviews shall satisfy DO-178C independence requirements:

| Review Type | Reviewer | Independence | Timing |
|-------------|----------|--------------|--------|
| Self-review | Author | N/A | Before PR submission |
| Peer code review | Developer who did not author the code | Required (DAL A) | During PR review, before merge |
| QA code audit | QA engineer | Required (DAL A) | Per QA sampling plan (SQAP) |

**Code review checklist items:**

- [ ] Code compiles without warnings on both platforms (GCC, `-Wall -Wextra -Werror`).
- [ ] Code complies with SCS-SF-2026-001.
- [ ] Code is traceable to low-level design elements.
- [ ] Code correctly implements the design.
- [ ] No dead code, no deactivated code without justification.
- [ ] Error handling is complete and consistent.
- [ ] Resource management follows RAII principles.
- [ ] Comments are accurate and sufficient.
- [ ] Unit tests accompany the code change with adequate coverage.
- [ ] Static analysis passes with zero findings of severity "error" or "warning."
- [ ] If LLM-assisted: AI-generation disclosure is present (see Section 13).

### 8.3 COTS / Reuse Assessment

#### 8.3.1 JSBSim Assessment

JSBSim v1.2.4 is classified as **COTS software** under DO-178C Section 12.3. The following assessment activities apply:

| Activity | Description | Status |
|----------|-------------|--------|
| Functional assessment | Verify JSBSim provides required 6-DOF functionality | To be performed |
| Interface assessment | Verify integration interfaces are well-defined and stable | To be performed |
| Error handling assessment | Verify JSBSim error behavior is documented and handled | To be performed |
| Version control | JSBSim v1.2.4 source pinned in project repository | Implemented |
| Change impact analysis | Any JSBSim update requires full regression and impact analysis | Policy established |
| Qualification | Assess whether JSBSim requires qualification per DO-178C 12.3 or additional verification | To be performed |

The COTS assessment shall determine one of the following dispositions:

1. **Accept with additional verification** — JSBSim is used as-is, with SynthFlight-side verification to confirm correct behavior for the specific use cases.
2. **Qualify as a verified component** — JSBSim source code is reviewed, tested, and qualified to the DAL A standard.
3. **Wrap and verify** — JSBSim is wrapped with runtime monitors that verify output validity.

#### 8.3.2 Other Reused Components

All test frameworks (Google Test, Catch2, Doctest, Boost.Test, CppUnit) are classified as **development tools**, not part of the deliverable software. They do not require DO-178C qualification unless their output is the sole means of verification (see DO-330 TQL assessment).

libFuzzer is classified as a **verification tool** and shall be assessed for tool qualification per DO-330.

## 9. Software Integration Process

### 9.1 Integration Strategy

SynthFlight follows a **bottom-up integration strategy**, building from foundational libraries to the full application:

```
Level 1: Math & Utility Libraries
    │
    ▼
Level 2: Core Models (Atmosphere ISA, Terrain Loaders, Sensor Models)
    │
    ▼
Level 3: Subsystems (JSBSim Integration, Flight Planning, Autopilot)
    │
    ▼
Level 4: System Services (Scenario Manager, Multi-Vehicle, Recording)
    │
    ▼
Level 5: Application (GUI, Vulkan Renderer, CLI/Batch Interface)
```

**Rationale:** Bottom-up integration enables early validation of mathematical foundations (atmospheric calculations, coordinate transformations, physics) before dependent subsystems are integrated. This approach is critical for a DAL A simulation product where mathematical accuracy is paramount.

### 9.2 Integration Levels — Detail

**Level 1 — Math & Utility Libraries**
- Linear algebra, matrix operations, quaternion arithmetic.
- Coordinate system transformations (ECEF, LLA, NED, body frame).
- Unit conversion utilities.
- Configuration file parsing.
- **Integration test:** Math libraries produce correct results for known-answer problems.

**Level 2 — Core Models**
- 1976 U.S. Standard Atmosphere (ISA) model implementation.
- Terrain data loaders (DTED L0/L1/L2, LiDAR point cloud, GeoTIFF DEM).
- Sensor models (EO/IR, SAR, LiDAR, GNSS error models).
- **Integration test:** Models produce validated outputs against reference data.

**Level 3 — Subsystems**
- JSBSim 6-DOF engine initialization, stepping, and state extraction.
- Flight planning (waypoint generation, pattern computation).
- Autopilot control law execution.
- **Integration test:** Subsystems interact correctly (e.g., atmosphere feeds JSBSim, terrain feeds collision detection).

**Level 4 — System Services**
- Scenario manager loads, configures, and executes simulation runs.
- Multi-vehicle coordination and deconfliction.
- Data recording captures time-stamped simulation state.
- **Integration test:** End-to-end scenario execution with data recording.

**Level 5 — Application**
- Vulkan renderer displays terrain and vehicle state.
- GUI provides mission planning and execution control.
- Batch/headless mode executes 1000+ scenarios unattended.
- **Integration test:** Full application smoke tests on both platforms.

         │           └──────────┘             │
         │                                    │
         ▼                                    ▼
    ┌──────────┐                        ┌──────────┐

    └──────────┘                        └──────────┘

### 12.3 DR Attributes

| Field | Description |
|-------|-------------|

| Severity | 1 (Critical), 2 (Major), 3 (Minor), 4 (Cosmetic) |

### 12.4 Severity Classification

| Severity | Definition | Response Time | Gate Impact |
|----------|------------|---------------|-------------|
| 1 — Critical | Safety impact, loss of function, incorrect simulation output affecting flight safety analysis | Immediate — work stops until triaged | Blocks all phase gates |
| 2 — Major | Significant functional defect, incorrect results, requirement not met | 5 business days to analyze | Blocks current phase gate |
| 3 — Minor | Minor functional issue, workaround exists, non-critical requirement partially met | 10 business days to analyze | Does not block gate (tracked) |
| 4 — Cosmetic | Documentation error, formatting, non-functional issue | Next scheduled maintenance | Does not block gate (tracked) |

### 12.5 DR Tracking

DRs shall be tracked using GitHub Issues on the project repository (github.com/vautieri/one-evening-pipeline) with the following conventions:

- Issue labels map to DR severity: `severity-1-critical`, `severity-2-major`, `severity-3-minor`, `severity-4-cosmetic`.
- Issue labels map to DR category: `cat-requirements`, `cat-design`, `cat-code`, `cat-test`, `cat-process`, `cat-tool`.
- Milestones map to iteration gates.
- DR closure requires verification evidence linked in the issue.

### 12.6 DR Metrics

The following metrics shall be tracked and reported at each iteration gate:

- Total DRs opened vs. closed per iteration.
- Open DR count by severity.
- DR aging (time from open to close).
- Defect density (DRs per KSLOC).
- Phase containment effectiveness (defects found in the phase introduced vs. later phases).

## 13. LLM-Assisted Development

### 13.1 Purpose

SynthFlight development may use LLM-based AI tools (specifically Goose and Claude) to assist with software development activities. This section establishes the policies, constraints, and documentation requirements for LLM-assisted development to maintain compliance with DO-178C DAL A objectives.

### 13.2 Scope of LLM Use

LLM tools may be used for the following activities:

| Activity | Permitted Use | Restrictions |
|----------|---------------|--------------|
| Requirements analysis | Assist with requirement decomposition, consistency checks | LLM output must be verified by a qualified engineer |
| Design exploration | Generate design alternatives, interface proposals | LLM output is advisory only — not a design authority |
| Code generation | Generate source code, unit tests, utility functions | All generated code subject to full review process |
| Code review assistance | Identify potential defects, suggest improvements | LLM findings require human confirmation |
| Test case generation | Generate test cases, test data, boundary conditions | All generated tests require human review and approval |
| Documentation | Generate documentation drafts, comments, descriptions | All documentation requires human review and approval |
| Static analysis triage | Assist with categorizing and analyzing static analysis findings | Final disposition by qualified engineer only |

LLM tools shall **not** be used for:

- Final approval or sign-off of any artifact.
- Safety assessment or hazard analysis conclusions.
- Certification credit without independent human verification.
- Replacement of required independent reviews (DO-178C independence).

### 13.3 Independence Considerations

**Critical:** LLM-generated artifacts do not satisfy the DO-178C independence requirement. Specifically:

1. **An LLM that generates code cannot serve as the independent reviewer of that same code.** If an LLM generates source code, the independent review must be performed by a qualified human engineer who did not use the LLM to perform the review.

2. **An LLM that generates test cases cannot provide independence for verification.** If an LLM generates test cases for code it also generated, those tests do not satisfy the independence objective. Independent verification must be performed by a different agent (human or a separately qualified process).

3. **LLM-assisted reviews are supplementary.** An LLM may assist a human reviewer by highlighting areas of concern, but the human reviewer retains full responsibility for the review findings.

4. **Common-mode concerns.** If the same LLM (same model, same session) generates both code and tests, a common-mode failure risk exists. Mitigation options:
   - Use different LLM sessions or models for code vs. test generation.
   - Require human-authored tests for safety-critical paths.
   - Apply additional verification techniques (formal analysis, diverse implementation comparison).

### 13.4 Review Requirements for LLM-Generated Artifacts

All artifacts generated with LLM assistance shall undergo the following review process:

| Step | Activity | Responsibility |
|------|----------|---------------|
| 1 | LLM generates artifact (code, test, document) | Developer using LLM tool |
| 2 | Developer reviews LLM output for correctness, completeness, and standard compliance | Originating developer |
| 3 | Developer marks artifact with LLM-generation disclosure (see Section 13.5) | Originating developer |
| 4 | Independent peer review (human, not LLM) | Independent reviewer |
| 5 | Standard phase review process (same as non-LLM artifacts) | Review team per Sections 6–8 |

**No reduction in review rigor is permitted for LLM-generated artifacts.** LLM-generated code receives the same (or greater) scrutiny as human-authored code.

### 13.5 Traceability of AI-Generated Artifacts

All LLM-generated or LLM-assisted artifacts shall be identified with the following metadata:

**For source code** (in file header or commit message):

```
// LLM-Assisted: Yes
// LLM-Tool: [Goose | Claude | Other — specify]
// LLM-Model: [Model identifier, e.g., Claude Opus 4.6]
// LLM-Session: [Date and session identifier if available]
// LLM-Scope: [Full generation | Partial assist | Review assist]
// Human-Reviewer: [Name of independent human reviewer]
```

**For commit messages:**

```
[LLM-Assisted] Description of change

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
LLM-Scope: [Full generation | Partial assist]
```

**For documents and plans:**

A section or metadata header indicating LLM involvement, the scope of assistance, and the human reviewer who approved the content.

### 13.6 LLM Tool Assessment

LLM tools used in SynthFlight development shall be assessed under the following framework:

| Assessment Area | Criteria |
|-----------------|----------|
| Tool identification | Tool name, version/model, access method (API, CLI, IDE plugin) |
| Functional scope | What the tool can and cannot do |
| Known limitations | Hallucination risk, training data cutoff, non-determinism |
| Qualification need | Assessment per DO-330 — is the tool's output directly used in certification evidence? |
| Mitigation | Review process, independence controls, verification of LLM output |

**DO-330 Tool Qualification Assessment:**

- If LLM output is **reviewed, modified, and verified by a human** before becoming a certification artifact, the LLM is classified as a **development aid** (no tool qualification required, but usage must be documented).
- If LLM output is **directly used as certification evidence without independent verification**, tool qualification to the appropriate TQL would be required. **This use case is not permitted under this SDP without prior approval from the DER.**

### 13.7 LLM Usage Metrics

The following metrics shall be tracked for LLM-assisted development:

| Metric | Purpose |
|--------|---------|
| Percentage of code lines generated with LLM assistance | Scope assessment |
| Percentage of test cases generated with LLM assistance | Scope assessment |
| Defect rate in LLM-generated code vs. human-authored code | Quality comparison |
| Review finding rate in LLM-generated artifacts | Process effectiveness |
| Number of LLM-generated artifacts rejected during review | Quality gate effectiveness |

These metrics shall be reported in the Software Accomplishment Summary (SAS) to provide transparency to the certification authority regarding the extent and effectiveness of LLM use in SynthFlight development.

## 1. Project Context

This project demonstrates **production-grade C++ unit testing** by exercising two
testable modules extracted from [JSBSim](https://github.com/JSBSim-Team/jsbsim),
an open-source flight dynamics model (FDM) widely used in aerospace, defense, and
simulation:

| Module | Classes | Domain |
|--------|---------|--------|
| **Math Library** | `FGColumnVector3`, `FGMatrix33` | 3D vectors, 3x3 matrices, Euler angle extraction, direction cosine matrices (DCMs) |
| **Atmosphere Model** | `FGStandardAtmosphere1976` | ISA 1976 U.S. Standard Atmosphere: temperature, pressure, density, speed of sound vs. altitude |

Both modules use **1-based indexing** following aerospace convention (v(1) = X,
v(2) = Y, v(3) = Z), and all internal units are Imperial (feet, Rankine,
slugs, psf) consistent with the JSBSim 6-DOF equations of motion.

**Why these modules?** They are self-contained, pure-computation libraries with
no external dependencies beyond the C++ standard library. They represent the
exact kind of numerically-sensitive code that defense/simulation teams must
regression-test rigorously: frame transforms, rotation matrices, atmosphere
lookups, and floating-point edge cases.

**The demonstration:** We test the *same production code* using **5 different
C++ unit test frameworks** so your team can evaluate each framework side-by-side
and pick the one that best fits your workflow, CI/CD pipeline, and coding
standards.

## 2. CRITICAL: Test Integrity Rules

Unit tests are only as valuable as the independence of their expected values.
The single most dangerous failure mode in any test suite — whether written by
a human or generated by AI — is **adjusting expected values to match code output
instead of investigating why the code disagrees with the reference.**

A test suite where 100% of tests pass is not inherently better than one where
98% pass and 2% have filed defects. The 2% of failures against published
reference data are often the most valuable output of the entire testing effort.

These three rules must be enforced in every code review that touches test
expected values.

### The Three Rules

#### Rule 1: Expected values come from an independent source, never from running the code

```cpp
// WRONG — derived by running the code and copying the output
EXPECT_NEAR(atm.GetStdPressure(alt), 476.1, 1e-2);  // "looks right"

// RIGHT — from published ISA 1976 table (NASA TM-X-74335, Table I)
// 36,000 ft geometric ≈ 35,938 ft geopotential
// Interpolated ISA value: P ≈ 476.13 psf (eq. 33a, L=-3.5662e-3 R/ft)
// Source: NASA TM-X-74335
EXPECT_NEAR(atm.GetStdPressure(36000.0), 476.13, 0.5);
```

If you cannot cite a published reference, a textbook, or a hand calculation for
the expected value, your test is a **tautology** — it only proves the code
returns what it returns.

#### Rule 2: When a test fails, triage — don't "fix"

```
When a test fails:
  1. Is the expected value from an independent reference?
     → The code has a bug. File it. Do NOT change the expected value.
  2. Is the expected value derived from running the code?
     → The test has a bug. Fix the test. Document why.
  3. Unsure?
     → Investigate. Don't touch either until you know.
```

A failing test against a published reference value is **the primary deliverable**
of a test suite. It is not an obstacle to overcome.

#### Rule 3: Separate reference tests from self-consistency tests

Your test suite should clearly distinguish two categories:

```cpp
// CATEGORY 1: REFERENCE TESTS
// Expected values from published sources (ISA 1976, textbooks, hand calcs).
// NEVER adjust expected values without a new source citation.
// These catch REAL BUGS in the code.
TEST(ISA_Reference, PressureAt32km)
{
    // Published: ISA 1976, Table I — P = 868.02 Pa = 18.13 psf at 32 km
    double geom_32km = atm.GeometricAltitude(104986.8766);
    EXPECT_NEAR(atm.GetStdPressure(geom_32km), 18.13, 0.1);
    // ↑ THIS WOULD HAVE CAUGHT THE BUG AT 27% ERROR
}

// CATEGORY 2: SELF-CONSISTENCY TESTS
// Verify internal mathematical identities (e.g., rho = P / (R * T)).
// Safe to adjust if the identity itself is correct.
// These catch CORRUPTION, not model errors.
TEST(ISA_Consistency, DensityEqualsP_over_RT)
{
    double rho   = atm.GetStdDensity(20000.0);
    double P     = atm.GetStdPressure(20000.0);
    double T     = atm.GetStdTemperature(20000.0);
    double rho_c = P / (R * T);
    EXPECT_NEAR(rho, rho_c, 1e-12);
    // ↑ Safe: both sides use the same code, so this tests internal coherence
}
```

Label your tests clearly. In code reviews, flag any change to a Category 1
expected value as requiring a source citation.

## 3. The 5 Frameworks -- Overview and Comparison

### Framework 1: Google Test (gtest)

- **Repository:** https://github.com/google/googletest
- **Version used:** v1.15.2 (fetched automatically via CMake `FetchContent`)
- **Description:** The most widely adopted C++ test framework. Provides
  `TEST()` and `TEST_F()` macros, a rich assertion set (`EXPECT_*` / `ASSERT_*`),
  automatic test discovery, parameterized tests, death tests, and built-in
  JUnit XML output for CI/CD integration. Includes Google Mock for mocking.

**Pros:**
- Industry standard; extensive documentation and community support
- `gtest_discover_tests()` CMake integration auto-registers every `TEST()` with CTest
- Rich assertion vocabulary: `EXPECT_DOUBLE_EQ`, `EXPECT_NEAR`, `EXPECT_THROW`
- JUnit XML output (`--gtest_output=xml:`) works out-of-the-box with Jenkins, GitHub Actions, Azure DevOps
- Supports test fixtures (`TEST_F`), typed tests, and value-parameterized tests
- Google Mock ships alongside for interface mocking

**Cons:**
- Requires building the gtest library (not header-only); adds ~30s to first configure
- Assertion macros use `EXPECT_*` / `ASSERT_*` naming that differs from `assert()`
- Compile times are moderate (not the fastest)

**Best for:** Large teams with corporate CI/CD standards, projects requiring JUnit XML
reports, shops that need mocking (Google Mock), and any team that values a mature,
well-supported ecosystem.

---

### Framework 2: Catch2

- **Repository:** https://github.com/catchorg/Catch2
- **Version used:** v3.7.1 (fetched automatically via CMake `FetchContent`)
- **Description:** A modern, expressive test framework with BDD-style
  `SECTION` / `SCENARIO` support and tag-based test filtering. Tests read like
  specifications. The v3 release is a compiled library (v2 was single-header).

**Pros:**
- Natural-language test names: `TEST_CASE("Vector3: cross product i x j = k", "[vector][arithmetic]")`
- Tag-based filtering: run `./test_math_catch2 "[matrix]"` to run only matrix tests
- `SECTION` blocks allow shared setup with branching test logic (no fixtures needed)
- `Catch::Approx` provides intuitive floating-point comparison: `REQUIRE(x == Approx(1.0).margin(1e-10))`
- `catch_discover_tests()` CMake integration auto-registers tests with CTest
- BDD macros (`SCENARIO` / `GIVEN` / `WHEN` / `THEN`) for requirements traceability

**Cons:**
- v3 is a compiled library (no longer single-header); v2 single-header is deprecated
- Slightly slower test discovery than gtest in very large suites
- No built-in mocking (use a separate library like Trompeloeil)

**Best for:** Rapid prototyping, teams that value readable test specifications,
projects where BDD-style tests align with requirements documents, and engineers
who prefer tag-based test selection over fixture hierarchies.

---

### Framework 3: doctest

- **Repository:** https://github.com/doctest/doctest
- **Version used:** v2.4.11 (fetched automatically via CMake `FetchContent`)
- **Description:** The fastest-compiling C++ test framework. API is deliberately
  similar to Catch2 (`TEST_CASE`, `SUBCASE`, `CHECK`, `REQUIRE`) but with
  minimal compile-time overhead. Can be embedded directly in production headers
  for in-situ testing.

**Pros:**
- Fastest compile times of any framework (claims 20-30x faster than Catch2 v2)
- `TEST_SUITE` blocks group related tests logically
- Minimal footprint: can be `#include`-d in production code and stripped at compile time
- API nearly identical to Catch2: low migration cost between the two
- `doctest::Approx` for floating-point comparison
- Single header option still available and fully supported

**Cons:**
- Smaller community than gtest or Catch2
- No built-in CMake test discovery helper (tests registered manually via `add_test()`)
- Fewer advanced features (no parameterized tests, no BDD macros)
- No built-in mocking

**Best for:** Embedded or real-time projects where compile time is critical,
codebases that want tests co-located with production headers, and teams migrating
from Catch2 who want faster builds without changing test syntax.

---

### Framework 4: Boost.Test

- **Repository:** https://www.boost.org/doc/libs/release/libs/test/
- **Description:** Part of the Boost C++ ecosystem. Provides `BOOST_AUTO_TEST_CASE`,
  `BOOST_CHECK_*`, and `BOOST_AUTO_TEST_SUITE` macros. Three usage modes:
  header-only, static library, and dynamic library.

**Pros:**
- Ships with Boost; no additional dependency if your project already uses Boost
- Mature and battle-tested (part of Boost since 2001)
- `BOOST_CHECK_CLOSE` for relative tolerance, `BOOST_CHECK_SMALL` for absolute tolerance
- `BOOST_DATA_TEST_CASE` for data-driven tests
- Integrates with Boost.Exception for structured exception testing
- Multiple output formats including XML

**Cons:**
- Requires Boost to be installed system-wide (not fetched by CMake automatically)
- Verbose macro names: `BOOST_CHECK_EQUAL`, `BOOST_AUTO_TEST_SUITE_END()`
- Documentation can be dense; learning curve for the full feature set
- Build configuration (header-only vs. dynamic) can be confusing

**Best for:** Teams already using Boost in their codebase, projects with
Boost-based infrastructure, and shops that prefer a single ecosystem for
libraries and testing.

---

### Framework 5: CppUnit

- **Repository:** https://freedesktop.org/wiki/Software/cppunit/
- **Description:** A JUnit-style xUnit framework for C++. Tests are methods on
  fixture classes derived from `CppUnit::TestFixture`, registered via
  `CPPUNIT_TEST_SUITE` / `CPPUNIT_TEST` macros. Requires a manual `main()`
  with a `TestRunner`.

**Pros:**
- Familiar to teams with Java/JUnit experience: fixture classes, setUp/tearDown, test suites
- Explicit test registration gives fine-grained control over test execution order
- `CPPUNIT_ASSERT_DOUBLES_EQUAL(expected, actual, delta)` for floating-point comparison
- Stable and predictable; no "magic" auto-discovery

**Cons:**
- Most boilerplate of all five frameworks (fixture class + macro registration + manual main)
- Requires system-installed CppUnit library (not fetchable via CMake)
- No auto-discovery: every test method must be listed in `CPPUNIT_TEST_SUITE`
- Smallest active community; limited modern C++ features
- No built-in mocking

**Best for:** Teams transitioning from Java/JUnit, legacy codebases already using
CppUnit, and projects where explicit OOP test structure is a requirement.

| Option | Default | Description |

### Step 5: Understand the Test Structure

This is the most important section for framework evaluation. Below, we show how
the **exact same test** -- "cross product: i x j = k" -- is written in each of
the five frameworks. This test verifies that the cross product of the unit X-axis
vector with the unit Y-axis vector produces the unit Z-axis vector, a fundamental
identity used in moment calculations (M = r x F) throughout any 6-DOF simulation.

#### Google Test

```cpp
#include <gtest/gtest.h>
#include "FGColumnVector3.h"
using namespace JSBSim;

static constexpr double EPS = 1e-10;

TEST(ColumnVector3, CrossProduct)
{
    // i x j = k
    FGColumnVector3 i(1.0, 0.0, 0.0);
    FGColumnVector3 j(0.0, 1.0, 0.0);
    FGColumnVector3 k = i * j;
    EXPECT_NEAR(k(1), 0.0, EPS);
    EXPECT_NEAR(k(2), 0.0, EPS);
    EXPECT_NEAR(k(3), 1.0, EPS);
}
```

**Key points:**
- `TEST(SuiteName, TestName)` -- two-level naming, no quotes needed for names
- `EXPECT_NEAR(actual, expected, tolerance)` -- continues on failure
- `ASSERT_NEAR(...)` variant would abort the test immediately on failure
- `EXPECT_DOUBLE_EQ` available for exact ULP-based comparison
- No `main()` needed: link with `GTest::gtest_main`

---

#### Catch2

```cpp
#include <catch2/catch_test_macros.hpp>
#include <catch2/catch_approx.hpp>
#include "FGColumnVector3.h"
using namespace JSBSim;
using Catch::Approx;

static constexpr double EPS = 1e-10;

TEST_CASE("Vector3: cross product i x j = k", "[vector][arithmetic]")
{
    FGColumnVector3 i(1.0, 0.0, 0.0);
    FGColumnVector3 j(0.0, 1.0, 0.0);
    FGColumnVector3 k = i * j;
    REQUIRE(k(1) == Approx(0.0).margin(EPS));
    REQUIRE(k(2) == Approx(0.0).margin(EPS));
    REQUIRE(k(3) == Approx(1.0).margin(EPS));
}
```

**Key points:**
- `TEST_CASE("descriptive name", "[tag1][tag2]")` -- human-readable names, tags for filtering
- `REQUIRE(expr)` aborts on failure; `CHECK(expr)` continues
- `Approx(value).margin(abs_tol)` or `.epsilon(rel_tol)` for float comparison
- Tags enable `./test_math_catch2 "[vector]"` to run all vector tests
- No `main()` needed: link with `Catch2::Catch2WithMain`

---

#### doctest

```cpp
#define DOCTEST_CONFIG_IMPLEMENT_WITH_MAIN
#include <doctest/doctest.h>
#include "FGColumnVector3.h"
using namespace JSBSim;

static constexpr double EPS = 1e-10;

TEST_SUITE("ColumnVector3") {

TEST_CASE("cross product i x j = k")
{
    FGColumnVector3 i(1.0, 0.0, 0.0);
    FGColumnVector3 j(0.0, 1.0, 0.0);
    FGColumnVector3 k = i * j;
    CHECK(k(1) == doctest::Approx(0.0).epsilon(EPS));
    CHECK(k(2) == doctest::Approx(0.0).epsilon(EPS));
    CHECK(k(3) == doctest::Approx(1.0).epsilon(EPS));
}

} // TEST_SUITE
```

**Key points:**
- `TEST_SUITE("name") { ... }` groups tests (similar to Catch2 tags but structural)
- `CHECK` continues on failure; `REQUIRE` aborts
- `doctest::Approx(value).epsilon(tol)` for float comparison
- `#define DOCTEST_CONFIG_IMPLEMENT_WITH_MAIN` provides `main()`
- Nearly identical syntax to Catch2 -- low migration cost between the two

#### Boost.Test

```cpp
#define BOOST_TEST_MODULE JSBSimMathTests
#include <boost/test/unit_test.hpp>
#include "FGColumnVector3.h"
using namespace JSBSim;

static constexpr double EPS = 1e-10;

BOOST_AUTO_TEST_SUITE(ColumnVector3)

BOOST_AUTO_TEST_CASE(cross_product)
{
    FGColumnVector3 i(1.0, 0.0, 0.0);
    FGColumnVector3 j(0.0, 1.0, 0.0);
    FGColumnVector3 k = i * j;
    BOOST_CHECK_CLOSE(k(3), 1.0, EPS);
    BOOST_CHECK_SMALL(k(1), EPS);
    BOOST_CHECK_SMALL(k(2), EPS);
}

BOOST_AUTO_TEST_SUITE_END()
```

**Key points:**
- `BOOST_AUTO_TEST_SUITE(name)` / `BOOST_AUTO_TEST_SUITE_END()` for grouping
- `BOOST_CHECK_CLOSE(actual, expected, percent_tolerance)` -- note: tolerance is a *percentage*
- `BOOST_CHECK_SMALL(value, absolute_tolerance)` -- checks value is near zero
- `BOOST_CHECK_EQUAL` for exact comparison
- `#define BOOST_TEST_MODULE` provides `main()`
- Test names must be valid C++ identifiers (underscores, not spaces)

#### CppUnit

```cpp
#include <cppunit/TestFixture.h>
#include <cppunit/extensions/HelperMacros.h>
#include <cppunit/ui/text/TestRunner.h>
#include "FGColumnVector3.h"
using namespace JSBSim;

static constexpr double EPS = 1e-10;

class ColumnVector3Test : public CppUnit::TestFixture {
    CPPUNIT_TEST_SUITE(ColumnVector3Test);
    CPPUNIT_TEST(testCrossProduct);
    CPPUNIT_TEST_SUITE_END();

public:
    void testCrossProduct()
    {
        FGColumnVector3 i(1.0, 0.0, 0.0);
        FGColumnVector3 j(0.0, 1.0, 0.0);
        FGColumnVector3 k = i * j;
        CPPUNIT_ASSERT_DOUBLES_EQUAL(0.0, k(1), EPS);
        CPPUNIT_ASSERT_DOUBLES_EQUAL(0.0, k(2), EPS);
        CPPUNIT_ASSERT_DOUBLES_EQUAL(1.0, k(3), EPS);
    }
};

CPPUNIT_TEST_SUITE_REGISTRATION(ColumnVector3Test);

int main()
{
    CppUnit::TextUi::TestRunner runner;
    runner.addTest(CppUnit::TestFactoryRegistry::getRegistry().makeTest());
    return runner.run() ? 0 : 1;
}
```

**Key points:**
- Inherits from `CppUnit::TestFixture` -- classic OOP pattern
- Every test method must be manually registered in `CPPUNIT_TEST_SUITE`
- `CPPUNIT_ASSERT_DOUBLES_EQUAL(expected, actual, delta)` -- note argument order
- Requires explicit `main()` with `TestRunner`
- `CPPUNIT_ASSERT_THROW(expr, exception_type)` for exception testing
- Most verbose of the five frameworks

Here is the pattern for adding a new test -- verifying that the dot product of

No other changes needed. `catch_discover_tests()` picks it up automatically.

TEST_CASE("dot product of orthogonal vectors is zero")

    FGColumnVector3 a(1.0, 0.0, 0.0);

BOOST_AUTO_TEST_CASE(dot_product_orthogonal_is_zero)

In `tests/cppunit/test_math_cppunit.cpp`:

**Step 1:** Add the method declaration in the `CPPUNIT_TEST_SUITE` block:

## 6. Using AI to Generate Tests

AI assistants (Claude, Copilot, ChatGPT, etc.) are effective at generating unit
test boilerplate, but they require explicit guardrails. By default, AI tools
optimize for **passing tests, not correct tests.** Without the right
instructions, an AI will adjust expected values, loosen tolerances, or remove
failing assertions to achieve a clean run — hiding the very defects your tests
exist to find. The following prompt template and process prevent this.

### The Prompt Template

When asking an AI to generate tests for your codebase, include this instruction:

> **"When a test fails, report the failure and diagnose whether the bug is in the
> test or the code. Do not modify expected values derived from published reference
> data. Failing tests against independent reference values are the primary output
> of this process — they are features, not bugs."**

### Step-by-Step Process for AI-Assisted Test Generation

```
1. GATHER REFERENCE DATA FIRST
   Before generating any tests, collect published reference values from
   authoritative sources (NASA reports, textbooks, MIL-STDs, manufacturer data).
   Feed these to the AI as the ground truth.

2. GENERATE TESTS WITH CITED SOURCES
   Ask the AI to generate tests where every expected value includes a comment
   citing its source. If the AI cannot cite a source, it must flag the expected
   value as "derived from code — needs independent verification."

3. RUN AND DO NOT TOUCH FAILURES
   Build and run. Any failures against cited reference values are likely real
   bugs. Do NOT let the AI "fix" them. Instead:
   - File the failure as a defect
   - Have the AI diagnose the root cause
   - Fix the production code, not the test

4. REVIEW SELF-CONSISTENCY TESTS SEPARATELY
   Tests that verify internal identities (e.g., A * A^-1 = I) can safely have
   their tolerances adjusted — they don't reference external truth.

5. HUMAN REVIEW ALL EXPECTED VALUES
   Before merging AI-generated tests, a domain expert must verify:
   - Are the reference values correct?
   - Are the unit conversions right? (The #1 source of errors)
   - Are the tolerances appropriate for the numerical method used?

6. VERIFY ≥ 85% PUBLIC API COVERAGE
   After the AI generates tests, audit the test suite against the source headers.
   List every public method, operator, and free function. Confirm each is called
   by at least one test. AI-generated tests tend to miss:
   - Compound assignment operators (+=, -=, *=, /=)
   - Accessor aliases (Entry() vs operator())
   - In-place mutators (T(), InitMatrix())
   - Error-handling paths (invalid enum, division by zero for /=)
   - Free functions (scalar * object, operator<<, Dump())
   - SL getter variants (GetDensitySL, GetSoundSpeedRatio, etc.)
   If coverage is below 85%, go back and explicitly request tests for the
   missing methods. Do not count the suite as complete until verified.
```

### What AI Does Well vs. What It Gets Wrong

| AI Excels At | AI Gets Wrong |
|---|---|
| Boilerplate: generating 30 test cases from an API quickly | **Adjusting expected values to match code output** |
| Mathematical property tests (associativity, commutativity, etc.) | **Unit conversions** (geopotential vs. geometric altitude!) |
| Edge cases: NaN, Inf, zero vectors, division by zero | **Published reference values** (may hallucinate or round) |
| Framework syntax: correct macros for gtest/Catch2/doctest/etc. | **Tolerance selection** (may use tight EPS where loose is needed) |
| Self-consistency tests (identities, round-trips) | **Distinguishing "test bug" from "code bug"** |

### The Golden Rule

> **A test suite where 100% pass is not inherently better than one where 98% pass
> and 2% have filed defects.** The 2% failures against reference data are the
> most valuable output of the entire testing effort. Never let anyone — human or
> AI — silence them without a root-cause investigation.

## 9. Test Coverage Summary

### Coverage Target: 85% of Public API

Every public method, operator, and free function in the source code must be called
by at least one test. The project targets **≥ 85% public API coverage** per source
file. This is verified by auditing every public method signature against test calls.

#### Mandatory coverage audit process

Before declaring a test suite complete, run this checklist:

1. **List every public method** in each header file (constructors, operators,
   accessors, free functions, error-handling paths).
2. **Grep the test files** for each method name. If a method is never called in
   any test, it is a coverage gap.
3. **Count tested vs. total** per file. Any file below 85% needs additional tests.
4. **Pay special attention to:**
   - Compound assignment operators (`+=`, `-=`, `*=`, `/=`) — easy to skip
   - Accessor aliases (`Entry()` vs `operator()`) — both must be exercised
   - Error-handling paths (division by zero, invalid enum, out-of-range index)
   - Free functions (`operator*(scalar, vector)`, `operator<<`, `DotProduct`)
   - In-place mutators (`T()`, `InitMatrix()`, `Normalize()`)

## Appendix B: Reference Bug Demonstration — ISA Pressure at High Altitude

This appendix documents a known defect in the extracted atmosphere model. It
serves as a **working example** of how reference tests catch real code defects,
and why the test integrity rules from Section 2 exist.

### The Defect

The extracted `FGStandardAtmosphere1976` model produces correct temperatures at
all ISA layer boundaries, but **incorrect pressures above 32 km (~105,000 ft)**.
The error compounds through successive layers and reaches ~27% by 47 km.

### Verification Data

| Altitude | Expected (ISA 1976) | Code Output | Error |
|----------|----------------------------|-------------|-------|
| Sea level | 2116.228 psf | 2116.228 psf | 0.00% |
| 11 km (36,089 ft) | 472.68 psf | 472.68 psf | 0.00% |
| 20 km (65,617 ft) | 114.35 psf | 114.34 psf | 0.01% |
| **32 km (104,987 ft)** | **18.13 psf** | **18.13 psf** | **< 0.01%** |
| **47 km (154,199 ft)** | **3.20 psf** | **2.32 psf** | **27.7%** |

### Root Cause

In `GetPressure()`, the method calls `GetTemperature()` (which includes bias and
gradient modifications) instead of `GetStdTemperature()` when computing the base
temperature `Tmb` for each layer. Under standard conditions (zero bias), this
works for most layers, but the gradient fadeout logic in `GetTemperature()`
introduces small floating-point deviations that accumulate through the iterative
pressure computation at higher layers.

```cpp
// In GetPressure() — current (incorrect for standard pressure):
double Tmb = GetTemperature(GeometricAltitude(BaseAlt));  // ← includes bias path

// Correct for standard pressure queries:
double Tmb = GetStdTemperature(GeometricAltitude(BaseAlt));  // ← pure ISA profile
```

The `CalculatePressureBreakpoints()` method has the same issue — it manually
reconstructs the temperature at each breakpoint using bias and gradient terms
rather than querying the standard profile directly.

### How a Reference Test Catches This

A properly structured reference test will fail immediately and surface the defect:

```cpp
TEST(ISA_Reference, PressureAt47km)
{
    FGStandardAtmosphere1976 atm;
    double geom_47km = atm.GeometricAltitude(154199.4751);

    // Published: ISA 1976, Table I — P = 110.906 Pa ≈ 3.204 psf at 47 km
    // Source: NASA TM-X-74335, Table I, row for H = 47 km
    EXPECT_NEAR(atm.GetStdPressure(geom_47km), 3.204, 0.1);
    // ↑ FAILS with actual = 2.32 → FILE AS DEFECT, DO NOT ADJUST
}
```

The correct workflow when this test fails:

1. **Do not change the expected value.** The 3.204 comes from NASA TM-X-74335.
2. **File a defect** against `FGStandardAtmosphere1976::GetPressure()`.
3. **Diagnose**: trace through the pressure computation at 47 km and identify
   where the biased temperature path introduces error.
4. **Fix the code**, not the test. Rerun. The test should now pass.

### Why This Example Matters

This is the exact kind of defect that is easy to hide:

- The error is zero at sea level and negligible below 20 km (where most flight
  sim scenarios operate), so basic tests pass
- The error only appears in layers that require chaining through multiple
  pressure breakpoints
- The output is a plausible-looking number (2.32 psf vs. 3.20 psf) — not NaN
  or zero — so it does not look obviously wrong without a reference value to
  compare against

A self-consistency test (`rho = P / (R * T)`) would pass at every altitude
because density, pressure, and temperature are all derived from the same
(incorrect) temperature path. **Only a reference test against published ISA data
catches this class of defect.**
